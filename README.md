# Ebaa Visits

نظام داخلي لإدارة زيارات عملاء المعارض، مبني باستخدام Next.js App Router وSupabase. الواجهة المرئية بالكامل عربية وRTL، بينما تبقى أسماء الملفات والجداول والمتغيرات باللغة الإنجليزية لتسهيل الصيانة البرمجية.

## ملخص النظام

- مصادقة حقيقية عبر Supabase Auth.
- صلاحيات حسب الدور مع حماية في الواجهة، الخادم، وسياسات RLS.
- لوحة تحكم متغيرة حسب الدور.
- إدارة العملاء والزيارات والمتابعات والإشعارات.
- تقارير تشغيلية ورسوم بيانية مبنية على بيانات Supabase.
- نموذج QR عام آمن لتسجيل اهتمام العميل داخل المعرض.
- إعدادات إدارية للمستخدمين والفروع والأقسام ومصادر العملاء وروابط QR.
- دعم الوضع الفاتح افتراضيًا مع زر تبديل للوضع الداكن.

## الصلاحيات

- `sales_employee`: يرى العملاء والزيارات والمتابعات المسندة له أو التي أنشأها، ويملك تقارير أداء شخصية.
- `branch_supervisor`: يرى بيانات فرعه فقط، ويمكنه الوصول إلى تقارير الفرع وروابط QR الخاصة به.
- `general_manager`: يرى البيانات والتقارير على مستوى كل الفروع، مع عرض إعدادات النظام دون صلاحيات إدارية كاملة.
- `admin`: يدير كل شيء، بما في ذلك المستخدمون والفروع والأقسام ومصادر العملاء.

تم تفعيل RLS على جميع الجداول، مع دوال مساعدة:

- `get_current_user_role()`
- `get_current_user_branch_id()`
- `is_admin()`
- `is_manager()`
- `is_branch_supervisor()`
- `is_sales_employee()`

## متغيرات البيئة

أنشئ ملف `.env.local` من `.env.example` ثم أضف القيم التالية:

```bash
NEXT_PUBLIC_SUPABASE_URL=
NEXT_PUBLIC_SUPABASE_ANON_KEY=

# يستخدم على الخادم فقط لإنشاء المستخدمين ونموذج QR العام.
SUPABASE_SERVICE_ROLE_KEY=
```

ملاحظات مهمة:

- لا تستخدم `SUPABASE_SERVICE_ROLE_KEY` داخل مكونات العميل.
- نموذج QR العام وإدارة المستخدمين تحتاجان إلى `SUPABASE_SERVICE_ROLE_KEY` على الخادم.
- في Vercel أضف المتغيرات نفسها من لوحة Project Settings ثم أعد النشر.

## إعداد Supabase

1. أنشئ مشروعًا جديدًا في Supabase.
2. انسخ `NEXT_PUBLIC_SUPABASE_URL` و`NEXT_PUBLIC_SUPABASE_ANON_KEY`.
3. من إعدادات API انسخ `service_role` وضعه في `SUPABASE_SERVICE_ROLE_KEY`.
4. طبّق الهجرات:

```bash
supabase db push
```

أو افتح SQL Editor في Supabase ونفّذ الملفات بالترتيب:

```text
supabase/migrations/001_initial_schema.sql
supabase/migrations/002_notifications_insert_own.sql
supabase/migrations/003_activity_logs_scoped_select.sql
```

الهجرة الأولى تنشئ الجداول، الفهارس، دوال الصلاحيات، سياسات RLS، والبيانات الأولية للفروع والأقسام ومصادر العملاء.

## بيانات الاختبار

لتجهيز بيئة QA واقعية، شغّل الملف:

الملف ينشئ حسابات اختبار للأدوار الأساسية، 5 فروع، 7 أقسام، 10+ مصادر عملاء، 50 عميلًا، 80 زيارة، 60 متابعة، سجلات نشاط، وإشعارات. جميع أسماء العملاء والفروع والبيانات التشغيلية الظاهرة عربية.

كلمة مرور حسابات الاختبار:


راجع خطة الاختبار التفصيلية في [TESTING.md](./TESTING.md).

## إنشاء أول مسؤول نظام

لأن إدارة المستخدمين محمية، أنشئ أول مستخدم من Supabase ثم أضف ملفه التعريفي يدويًا:

1. من Supabase Auth أنشئ مستخدمًا بالبريد وكلمة المرور.
2. انسخ قيمة `id` للمستخدم من جدول `auth.users`.
3. نفّذ SQL مشابهًا:

```sql
insert into public.profiles (
  id,
  full_name,
  email,
  phone,
  role,
  branch_id,
  is_active
) values (
  'USER_ID_FROM_AUTH',
  'مسؤول النظام',
  'admin@example.com',
  null,
  'admin',
  null,
  true
);
```

بعد ذلك سجّل الدخول من `/login` واستخدم صفحة `/settings/users` لإنشاء بقية المستخدمين.

## النشر على Vercel

1. ارفع المشروع إلى مستودع Git.
2. أنشئ مشروعًا جديدًا في Vercel واختر المستودع.
3. أضف متغيرات البيئة:
   - `NEXT_PUBLIC_SUPABASE_URL`
   - `NEXT_PUBLIC_SUPABASE_ANON_KEY`
   - `SUPABASE_SERVICE_ROLE_KEY`
4. تأكد أن إعدادات Supabase Auth تسمح بعنوان النشر:
   - Site URL
   - Redirect URLs
5. نفّذ الهجرات على قاعدة الإنتاج قبل استخدام النظام.
6. أنشئ أول مسؤول نظام كما هو موضح أعلاه.

## اختبار الأدوار

- موظف مبيعات: أنشئ عميلًا وزيارة ومتابعة، وتأكد أنه لا يرى بيانات موظف آخر.
- مشرف فرع: تأكد أن الجداول والتقارير تعرض فرعه فقط، وأن صفحة روابط QR تعرض فروعه المسموحة.
- مدير عام: تأكد أن التقارير ولوحة التحكم تعرض كل الفروع دون صلاحيات تعديل المستخدمين.
- مسؤول النظام: جرّب إضافة مستخدم، تعديل فرع، تعطيل قسم، وإضافة مصدر عميل.
- مستخدم غير نشط: عطّل `is_active` في `profiles` وتأكد أنه لا يستطيع دخول الصفحات الداخلية.

## اختبار تدفق البيانات

- إنشاء عميل جديد من `/customers/new`.
- إدخال رقم جوال مكرر والتأكد من ظهور خيار فتح ملف العميل أو تسجيل زيارة له.
- تسجيل زيارة من `/visits/new` والتأكد من تحديث حالة العميل واحتمالية الشراء.
- إضافة موعد متابعة من نموذج الزيارة والتأكد من إنشاء سجل في `follow_ups`.
- إكمال متابعة من `/follow-ups/[id]` مع إنشاء متابعة جديدة عند الحاجة.
- فتح نموذج `/qr/[branchId]` وإرسال اهتمام عميل، ثم التأكد من إنشاء عميل أو تحديثه وإنشاء زيارة QR وإشعار للمشرف.
- فتح `/dashboard` و`/reports` للتأكد من تحديث الأرقام حسب الصلاحيات.

## ملاحظات إنتاجية

- التصدير الحالي يدعم CSV. يمكن إضافة Excel لاحقًا بمكتبة مخصصة مثل `xlsx`.
- التقارير الحالية تضبط نطاق البيانات وتطبق الصلاحيات، ومع نمو البيانات جدًا يفضل نقل بعض التجميعات إلى دوال SQL أو Views.
- لا يوجد مزود رسائل SMS أو WhatsApp مدمج بعد.
- سجل النشاط يغطي العمليات الأساسية، ويمكن توسيعه لاحقًا ليشمل كل تغييرات الإعدادات.
- نموذج QR يستخدم حماية الخادم وRLS، لكن يمكن إضافة rate limiting خارجي عند النشر.
